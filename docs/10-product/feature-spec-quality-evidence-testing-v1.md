# Feature Spec: Quality Evidence v1 (Testing First)

## Summary

Add a repo-native quality evidence subsystem that makes the current testing
state of a repository understandable from local artifacts alone.

V1 stays narrow:

- authored testing intent remains a local contract under `docs/30-contracts/`
- observed testing evidence is normalized into JSON and Markdown under
  `artifacts/quality/testing/`
- Workbench exposes a small command surface centered on `quality sync` and
  `quality show`

The goal is trustworthy evidence, not a general-purpose CI/CD layer.

## Goals

- Make the current testing state understandable for humans and AI agents from
  the repository alone.
- Separate authored truth from observed truth so green results do not replace
  intentional engineering judgment.
- Normalize .NET test inventory, TRX results, and coverage into stable JSON
  contracts.
- Generate human-readable summaries alongside machine-readable artifacts.
- Reuse existing Workbench operating patterns: local artifacts as canonical,
  sync/show command pairs, JSON envelopes, and generated repo views.
- Create a clean path to later analyzer, mutation, fuzz, and AI-assisted
  evidence without over-designing V1.

## Non-goals

- Turning Workbench into a generic CI/CD orchestrator or pipeline manager.
- Treating dashboards, hosted systems, or transient CI logs as the canonical
  source of truth.
- Autonomous test authoring or autonomous evidence remediation.
- Semantic judgment about whether a test is "good".
- Merge gating, policy enforcement, or required-status integration.
- Flaky-test intelligence, test-history warehousing, or trend databases.
- Broad multi-language normalization beyond what is needed for a clean .NET-led
  extension point.

## User stories / scenarios

- As a repo maintainer, I can see what tests exist, what most recently ran, and
  whether the current evidence matches the repo's stated testing expectations.
- As a reviewer, I can open one generated Markdown summary and understand
  testing scope, failures, coverage posture, and obvious evidence gaps without
  re-running tooling.
- As an AI agent, I can read normalized JSON artifacts and reason about the
  current testing state without scraping TRX or Cobertura files directly.
- As a spec/ADR author, I can state what matters, what is intentionally not
  covered yet, and what confidence is expected without that authored intent
  being overwritten by observed results.

## Operating model

### Authored truth

V1 keeps authored testing intent in local docs/contracts. The compatibility
anchor is the existing `docs/30-contracts/test-gate.contract.yaml`, expanded
from a pure gate file into a broader testing-intent contract.

Authored truth answers:

- what behaviors or code areas matter
- what risks are notable
- what kinds of tests are expected
- what confidence target is desired
- what is intentionally untested for now
- what docs, work items, and code refs explain the intent

### Observed truth

Observed truth is generated by Workbench from current repo evidence and written
under `artifacts/quality/testing/`.

Observed truth answers:

- what tests were discovered
- what tests ran
- what passed, failed, or skipped
- how long runs took
- what coverage was observed
- what gaps are detectable from current evidence alone

Observed truth never overwrites authored truth. The quality report compares the
two and reports mismatches, but it does not change the authored contract.

Layout rules:

- `docs/30-contracts/test-gate.contract.yaml` remains the human-authored source
  for testing intent in V1 to preserve compatibility.
- JSON schemas live beside other Workbench contracts under `docs/30-contracts/`.
- Current observed truth uses stable filenames under
  `artifacts/quality/testing/`; V1 does not introduce a history database.
- `quality-summary.md` is generated, human-readable, and derived entirely from
  the JSON artifacts plus authored contract inputs.

## Requirements

### V1 evidence flow

1. `workbench quality sync` reads the authored testing-intent contract.
2. It discovers the current test inventory for the repo.
3. It ingests existing test-result artifacts from `dotnet test` outputs.
4. It ingests existing coverage artifacts from Cobertura-compatible outputs.
5. It normalizes those inputs into stable JSON artifacts.
6. It generates a quality report JSON artifact plus a Markdown summary.
7. It returns a structured JSON envelope and human-readable summary output.

### Required V1 outputs

- `test-inventory.json` records discovered tests and project-level discovery
  summaries.
- `test-run-summary.json` records normalized run outcomes from TRX evidence.
- `coverage-summary.json` records normalized coverage results.
- `quality-report.json` compares authored truth to observed truth and records
  detectable gaps.
- `quality-summary.md` provides a readable summary for humans and reviewers.

### Required V1 comparisons

The quality report must be able to flag at least:

- missing run evidence for discovered test projects
- missing coverage evidence for scoped critical files
- coverage below authored thresholds
- authored required tests that are absent from inventory or absent from the
  latest observed run
- intentionally untested areas that remain unresolved but documented

V1 must not claim to judge whether a test is high quality. It only reports what
the authored contract asked for and what the observed evidence shows.

### Linking requirements

Where practical, V1 should carry links from authored truth into the quality
report:

- related docs
- related work items
- related files or code refs

This lets reviewers and agents jump from evidence to rationale.

### Raw artifact handling

Raw TRX and Cobertura files are inputs, not the canonical observed record.
Workbench may reference their paths in the normalized JSON, but the stable
observed truth for the subsystem is the normalized artifact set under
`artifacts/quality/testing/`.

## Command surface

V1 keeps the surface deliberately small.

### `workbench quality sync`

Purpose:
- discover testing inventory
- ingest test-result and coverage artifacts
- write normalized observed-truth artifacts
- generate the quality report

Core options:
- `--contract <path>`: defaults to `docs/30-contracts/test-gate.contract.yaml`
- `--results <path>`: TRX file or directory root to ingest
- `--coverage <path>`: Cobertura file or directory root to ingest
- `--out-dir <path>`: defaults to `artifacts/quality/testing`
- `--dry-run`

JSON `data` shape:

```json
{
  "inventory": {
    "path": "artifacts/quality/testing/test-inventory.json",
    "projects": 2,
    "tests": 84
  },
  "results": {
    "path": "artifacts/quality/testing/test-run-summary.json",
    "runId": "20260307T000000Z",
    "status": "failed",
    "passed": 80,
    "failed": 3,
    "skipped": 1
  },
  "coverage": {
    "path": "artifacts/quality/testing/coverage-summary.json",
    "available": true,
    "lineRate": 0.82,
    "branchRate": 0.74
  },
  "report": {
    "jsonPath": "artifacts/quality/testing/quality-report.json",
    "markdownPath": "artifacts/quality/testing/quality-summary.md",
    "status": "warn",
    "findings": 4
  },
  "dryRun": false
}
```

### `workbench quality show`

Purpose:
- render the latest normalized report or a selected artifact
- avoid direct parsing of raw tool outputs by humans or agents

Core options:
- `--kind <report|inventory|results|coverage>`: defaults to `report`
- `--path <path>`: optional explicit artifact path

Behavior:
- default table output shows the current quality report summary
- `--format json` returns the selected normalized artifact in the standard
  envelope

No extra V1 subcommands are required.

## UX notes

- The default `quality sync` table output should read like a repo health brief,
  not a CI console dump.
- `quality-summary.md` should present authored truth, observed truth, and gaps
  in separate sections so readers do not confuse intent with evidence.
- `quality show` should default to the latest report, because that is the
  decision surface humans and agents usually need first.

## Dependencies

- A .NET solution or repo with discoverable test projects.
- Test results emitted in TRX format or a cleanly mappable equivalent.
- Coverage results emitted in Cobertura-compatible XML or a cleanly mappable
  equivalent.
- Existing repo-native contract management under `docs/30-contracts/`.
- Existing Workbench JSON output envelopes and sync/show command conventions.

## Phased plan

### Phase 1: inventory + results + coverage

- Expand `test-gate.contract.yaml` into the authored testing-intent contract.
- Discover test projects and test cases, then emit `test-inventory.json`.
- Ingest TRX results and emit `test-run-summary.json`.
- Ingest Cobertura coverage and emit `coverage-summary.json`.
- Generate `quality-report.json` and `quality-summary.md`.
- Expose `workbench quality sync` and `workbench quality show`.

### Phase 2: analyzer/static-check evidence + changed-file heuristics

- Add analyzer/static-check evidence as another observed evidence kind.
- Record changed-file heuristics to highlight files touched without nearby
  evidence.
- Extend the quality report to show code-analysis signals beside testing
  evidence without treating them as the same thing.

### Phase 3: mutation/fuzz evidence

- Add optional mutation and fuzz evidence contracts plus normalized outputs.
- Keep execution orchestration out of scope; Workbench ingests results rather
  than owning the tools.
- Extend the report to show where advanced evidence exists or is absent.

### Phase 4: AI-assisted remediation and policy suggestions

- Add optional AI-authored suggestions that point to missing evidence or likely
  remediation options.
- Keep suggestions advisory and repo-native.
- Continue to separate observed evidence from any proposed enforcement or
  workflow automation.

## Risks and mitigations

- Risk: the existing `test-gate.contract.yaml` becomes overloaded.
  - Mitigation: keep V1 compatible, but structure it as authored testing intent
    and reserve a later rename if the broader role proves stable.
- Risk: test discovery varies across frameworks and adapters.
  - Mitigation: make discovery method explicit in inventory output and surface
    warnings instead of pretending all tests were discovered.
- Risk: normalized evidence drifts from raw inputs.
  - Mitigation: record source input paths, generation time, and tool metadata in
    every observed artifact.
- Risk: users interpret the report as a gate.
  - Mitigation: keep status vocabulary about evidence sufficiency, not merge
    permission, and omit policy-enforcement features from V1.
- Risk: stable filenames lose prior snapshots.
  - Mitigation: rely on git history for auditability in V1; do not add a
    dedicated history store yet.
